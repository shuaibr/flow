<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| env → process | `FLOW_DB_KEY` (the entire security of the encrypted store) crosses from the environment into the process; must never be persisted to repo/config/log |
| process → disk | Encrypted DB pages + WAL/shm sidecars written to `data/`; on-disk artifacts must be ciphertext and gitignored |
| pydantic schema → DB payload | Untrusted/raw payload shapes validated before becoming a stored JSON payload |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Information Disclosure | `FLOW_DB_KEY` in repo/config | mitigate | Key read from env only in cli.py; `.gitignore` covers `.env`; `.env.example` holds placeholders only; acceptance test asserts `git ls-files .env` is empty |
| T-01-02 | Information Disclosure | `data/flow.db` + `-wal`/`-shm` sidecars on disk | mitigate | SQLCipher full-DB AES-256 via PRAGMA-key-first factory; `.gitignore` excludes `data/`, `*.db`, `*.db-wal`, `*.db-shm`; ciphertext asserted by grep-returns-nonzero test |
| T-01-03 | Tampering | PRAGMA key issued outside the factory (unkeyed access vector) | mitigate | Single enforcement point asserted by `grep -c "PRAGMA key" == 1`; no PRAGMA key in cli.py/schema.py |
| T-01-04 | Tampering | malformed payload reaching the store | mitigate | pydantic `HRVStatusPayload` validates + `field_validator` rejects out-of-range values before any DB write |
| T-01-SC | Tampering | pip/uv installs (supply chain) | mitigate | All 7 packages slopcheck-[OK] in RESEARCH Package Legitimacy Audit; pinned `uv.lock`; no `--upgrade`; no [ASSUMED]/[SUS] packages so no blocking human checkpoint required |
</threat_model>
